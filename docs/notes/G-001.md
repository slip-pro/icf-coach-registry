## Context

First sprint of the project. Built from scratch — no prior code.
Source materials: product concept (temp/), HTML mockup, ICF brandbook.
Architecture: standalone JS widget + Google Sheets backend, embedded in WordPress page.

## Key Decisions

- D-001: Google Sheets as data backend (not WordPress CPT)
- D-002: Standalone JS widget (not WP plugin)
- D-003: Claude Code as primary developer
- DESIGN.md extracted from mockup by @designer before coding started
- ES modules (import/export), no build tools, no npm
- CSS isolation via `icf-` BEM prefix
- Mock data fallback when no Google Sheet ID configured

## Audit Results

- 0 CRITICAL, 3 HIGH (all fixed), 10 MEDIUM (2 fixed, 8 deferred as TD), 8 LOW
- Key fixes: URL injection prevention, email sanitization, memory leak fix, focus-visible styles, fetch timeout

## Problems & Solutions

- **Parallel agent conflict**: T4 (cards.js) and T6 (contacts.js) both implemented contact rendering. Resolved in T7 by making cards.js import from contacts.js.
- **PDF reader unavailable**: Could not read ICF brandbook PDF (poppler not installed). Used HTML mockup as design source instead.

## Patterns

- Parallel development of independent components works well (T4/T5/T6)
- @designer before @developer for UI projects — produced clean DESIGN.md that all developers followed
- Audit after development catches real issues (URL injection was a genuine HIGH)

## Reflection

1. **What was hard**: Integrating work from parallel agents — different naming conventions for contact block between cards.js and contacts.js.
2. **Decisions**: ES modules without build tools keeps deployment simple for non-developer owner. Trade-off: no minification, no tree-shaking.
3. **Instructions**: No changes needed — the workflow worked smoothly.
